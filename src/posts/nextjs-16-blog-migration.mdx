# Next.js 16으로 블로그를 마이그레이션한 이야기

작년에 Next.js 13에서 15로 마이그레이션하면서 꽤 고생했던 기억이 있는데, 이번에는 15에서 16으로 올리면서 동시에 블로그 전체를 리뉴얼했습니다.
솔직히 말하면 Next.js 버전 업 자체보다는 주변 도구들을 갈아엎는 게 더 큰 작업이었습니다.

이번 글에서는 마이그레이션 과정에서 겪은 것들과 결정들에 대해서 정리해보겠습니다.

## 마이그레이션 범위

이번에 바꾼 것들을 정리하면 이렇습니다.

| 항목 | Before | After |
|:-----|:-------|:------|
| Framework | Next.js 15.2.8 | Next.js 16.2.0 |
| Lint & Format | ESLint + Prettier | Biome |
| Package Manager | npm | pnpm |
| Testing | 없음 | Vitest + Testing Library |
| CI/CD | 없음 | GitHub Actions + Dependabot + CodeQL |
| Release | 수동 | release-please 자동화 |
| Design | 기본 shadcn | Indigo-Violet 테마 + 전면 리뉴얼 |

하나씩 어떤 과정을 거쳤는지 정리하겠습니다.

## Next.js 15 → 16 업그레이드

### 걱정했던 것들

Next.js 16은 2025년 10월에 정식 릴리즈됐습니다. 메이저 버전 업이라 Breaking Change가 있을 거라 걱정했는데, 저의 경우에는 생각보다 수월했습니다.

Next.js 16의 주요 변경점은 이렇습니다.

- **비동기 Request API 완전 필수화** — `cookies()`, `headers()`, `params`, `searchParams`의 동기 접근이 완전히 제거됨
- **Turbopack 기본화** — dev와 build 모두 Turbopack이 기본 번들러
- **React Compiler 내장** — 자동 메모이제이션

### 실제로 겪은 문제

다행히 저는 Next.js 15에서 이미 `await params`, `await searchParams` 패턴을 사용하고 있었기 때문에, 비동기 API 관련해서는 수정할 게 없었습니다.

유일하게 겪은 문제는 CI에서 타입 체크가 실패한 것이었습니다.

```typescript title="에러 내용"
error TS2307: Cannot find module '../../../../public/images/chahyunwoo-profile.jpg'
```

Next.js 16에서 `next-env.d.ts`의 구조가 바뀌면서, CI에서 빌드 전에 `tsc`를 돌리면 이미지 타입 선언을 못 찾는 문제였습니다. 해결은 간단했는데, CI에서 빌드를 먼저 실행하고 타입 체크를 나중에 돌리는 것으로 순서를 바꿨습니다.

```yaml title=".github/workflows/ci.yml"
- name: Build
  run: pnpm build

- name: Type Check  # 빌드 후에 실행
  run: pnpm tsc --noEmit
```

## ESLint + Prettier → Biome

이건 사실 Next.js 마이그레이션보다 더 큰 작업이었습니다.

### 왜 바꿨나

회사 프로젝트에서 이미 Biome를 사용하고 있었고, lint와 format을 하나의 도구로 통합할 수 있다는 점이 매력적이었습니다. ESLint + Prettier 조합은 설정 파일이 여러 개 필요하고, 가끔 서로 충돌하기도 하거든요.

### 전환 과정

1. ESLint, Prettier, 관련 패키지 전부 제거
2. Biome 설치 및 설정
3. `biome check --write .`로 전체 코드 포맷 변환

포맷이 바뀌면서 전체 파일의 quote 스타일(`"` → `'`), 세미콜론 제거, trailing comma 등이 한번에 변경되었습니다.

<Callout type="warning">
Biome로 전환하면 거의 모든 파일이 변경됩니다. git blame이 무의미해질 수 있으니 `.git-blame-ignore-revs`에 해당 커밋을 등록하는 것을 추천합니다.
</Callout>

주의할 점은 Biome의 린트 룰이 ESLint보다 엄격한 부분이 있다는 것입니다. 저의 경우 `noForEach`, `noArrayIndexKey`, `noDangerouslySetInnerHtml` 등에서 많은 경고가 발생했습니다. JSON-LD 삽입에 필수적인 `dangerouslySetInnerHTML`은 biome.json에서 off로 설정했고, 나머지는 코드를 수정하거나 warn으로 처리했습니다.

## npm → pnpm

### 전환 이유

디스크 절약과 설치 속도도 있지만, 가장 큰 이유는 **의존성 관리의 엄격함**이었습니다.

npm은 hoisting으로 인해 `package.json`에 명시하지 않은 패키지도 사용할 수 있는 경우가 있습니다. pnpm은 이걸 허용하지 않아서, 실제로 전환하면서 숨어있던 문제가 드러났습니다.

```bash
# 이 에러가 pnpm 전환 후 발생
error TS2307: Cannot find module 'fast-glob'
```

코드에서 `fast-glob`을 import하고 있었는데, `package.json`에는 `glob` 패키지만 있었습니다. npm에서는 `glob`의 하위 의존성으로 설치된 `fast-glob`을 가져다 쓸 수 있었지만, pnpm에서는 불가능했습니다. `fast-glob`을 직접 의존성으로 추가하고, 사용하지 않는 `glob`은 제거했습니다.

## 테스트 환경 구축

블로그에 테스트가 필요한가 싶었지만, CI에서 빌드 전에 유틸 함수나 서비스 로직을 검증하는 게 생각보다 유용했습니다.

```typescript title="src/__tests__/lib/utils.test.ts"
describe('formatDate', () => {
  it('should format date in English', () => {
    const result = formatDate('2025-01-15')
    expect(result).toContain('Jan')
    expect(result).toContain('15')
    expect(result).toContain('2025')
  })
})

describe('estimateReadingTime', () => {
  it('should return at least 1 minute', () => {
    expect(estimateReadingTime('short')).toBe(1)
  })
})
```

Vitest를 선택한 이유는 Vite 기반이라 빠르고, Jest와 거의 동일한 API를 제공하기 때문입니다. `@testing-library/react`와 함께 사용하면 컴포넌트 테스트도 가능합니다.

## CI/CD 자동화

이전에는 아무런 자동화가 없었습니다. 이번에 추가한 것들은 다음과 같습니다.

- **GitHub Actions CI** — PR마다 Biome lint, type-check, Vitest, build 자동 실행
- **CodeQL** — 코드 보안 취약점 자동 분석
- **Lighthouse CI** — PR마다 성능/접근성/SEO 점수 감사
- **Dependabot** — npm, GitHub Actions 의존성 자동 업데이트
- **release-please** — 커밋 메시지 기반 CHANGELOG.md, GitHub Release 자동 생성
- **Branch Protection** — main 브랜치에 PR + CI 통과 필수

<Callout type="tip">
`release-please`를 사용하면 `feat:` 커밋은 minor 버전, `fix:` 커밋은 patch 버전으로 자동 올라갑니다. `BREAKING CHANGE`를 포함하면 major 버전이 올라갑니다.
</Callout>

## 디자인 리뉴얼

기존 블로그는 기본 shadcn 테마에 가까웠는데, 이번에 Indigo-Violet 기반의 커스텀 테마로 전면 변경했습니다.

### 컬러 테마

라이트/다크 모드 모두 oklch 색공간 기반으로 설계했습니다. primary 컬러를 indigo-violet 계열로 통일하면서, MDX의 Callout(blue, green, amber, purple), Highlight(fuchsia), 인라인 코드(orange) 등과 어울리도록 조정했습니다.

### 레이아웃 변경

| 영역 | Before | After |
|:-----|:-------|:------|
| 사이드바 | 서브태그 트리 (Collapsible) | 카테고리 아이콘 + 태그 클라우드 + 최근 포스트 |
| 포스트 카드 | 기본 Card | 호버 효과, 읽기 시간, 태그 툴팁 |
| 포스트 상세 | 상단 접기형 TOC | 우측 sticky TOC + 모바일 바텀시트 |
| 검색 | 없음 | Cmd+K 커맨드 팔레트 |
| About | 카드 나열 | 타임라인 경력 + 프로젝트 카드 |

### 검색 기능

`cmdk` 기반의 커맨드 팔레트를 구현했습니다. Zustand 스토어로 검색 상태를 관리하고, 200ms 디바운스를 적용했습니다. 카테고리별 그룹핑도 지원합니다.

### 모바일 TOC

PC에서는 우측 sticky TOC가 따라다니지만, 모바일에서는 공간이 부족합니다. 우하단에 플로팅 버튼을 배치하고, 누르면 바텀시트로 목차가 올라오도록 구현했습니다. IntersectionObserver로 현재 읽고 있는 섹션을 하이라이트합니다.

## 성능 최적화

Lighthouse 점수를 최대한 끌어올리는 작업을 진행했습니다.

### 주요 최적화 항목

- **데드코드 제거** — 미사용 컴포넌트 5개, 패키지 2개(recharts, tailwindcss-animate), 함수 삭제
- **이미지 최적화** — AVIF/WebP 포맷, fetchPriority, responsive sizes
- **폰트 CLS 제거** — Pretendard fallback 폰트 정의 (`size-adjust`, `ascent-override`)
- **접근성** — aria-label, 터치 타겟 44px 보장, 색상 대비 개선
- **CSS 최적화** — `optimizeCss` 실험적 기능 활성화

### 결과

| Category | Score |
|:---------|:------|
| Performance | **99** |
| Accessibility | **95** |
| Best Practices | **100** |
| SEO | **100** |

<Callout type="info">

## 정리

### 좋았던 점

- Biome 도입으로 lint/format 설정이 단순해짐
- pnpm 전환으로 숨어있던 의존성 문제를 발견
- CI/CD 자동화로 머지 전 품질 보장
- release-please로 릴리즈 관리 자동화
- Lighthouse 99/95/100/100 달성

### 고려할 점

- Biome 전환 시 거의 모든 파일이 변경되므로 별도 커밋으로 분리하는 게 좋음
- Next.js 16의 `next-env.d.ts` 변경으로 CI 순서 조정이 필요할 수 있음
- Vercel의 dynamic 페이지는 `Cache-Control: no-store`가 설정되어 bfcache 불가
- 로컬 Lighthouse 점수와 실제 배포 점수는 크게 다를 수 있음
